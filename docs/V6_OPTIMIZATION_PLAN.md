## 四、串关模块重构（核心重点）

### 4.1 串关筛选器（独立模块）

**核心原则**：串关有独立的筛选逻辑，不复用单场筛选器。

```
┌─────────────────────────────────────────────┐
│           串关专用筛选器                      │
├─────────────────────────────────────────────┤
│                                             │
│  输入: 所有比赛的全市场预测                   │
│       ├── SPF (胜平负)                       │
│       ├── RSPF (让球胜平负)                  │
│       ├── Totals (总进球)                    │
│       ├── Double Chance (双选)               │
│       └── Cross Market (跨市场)              │
│                                             │
│  筛选: 比单场更宽松的阈值                    │
│       ├── 单腿价值 ≥ 2% (单场是5%)           │
│       ├── 单腿概率 ≥ 20%                     │
│       ├── 单腿赔率 1.20 ~ 不设上限           │
│       └── 每场保留最优2-3个选项              │
│                                             │
│  组合: 生成2串1/3串1                         │
│       ├── 跨联赛优先                         │
│       ├── 跨玩法优先                         │
│       ├── 组合赔率 2.0 ~ 不设上限            │
│       └── 组合概率 ≥ 5%                      │
│                                             │
│  输出: 串关投注池                            │
└─────────────────────────────────────────────┘
```

### 4.2 腿类型设计

| 腿类型 | 说明 | 概率计算 | 赔率来源 |
|--------|------|----------|----------|
| **单选腿** | 一个市场一个选项 | 模型直接输出 | 市场赔率 |
| **双选腿** | 一个市场两个选项 | P(选项1)+P(选项2) | 市场或计算 |
| **跨市场腿** | 同场不同市场各一 | P(市场1)×P(市场2)×相关系数 | 赔率相乘 |

**双选赔率计算**：
- 胜平(1X) = 1 / (1/odds_胜 + 1/odds_平)
- 平负(X2) = 1 / (1/odds_平 + 1/odds_负)
- 胜负(12) = 1 / (1/odds_胜 + 1/odds_负)

**跨市场相关系数**：
- SPF平 + RSPF平 → 0.85
- SPF胜 + Totals大 → 0.60
- SPF胜 + Totals小 → 0.40

### 4.3 串关参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大长度 | 2-3腿 | 2串1为主 |
| 最低组合赔率 | 2.0 | 确保回报 |
| 最高组合赔率 | 不设上限 | 允许高赔存在 |
| 单腿最低概率 | 20% | 放宽，允许高赔腿 |
| 组合最低概率 | 5% | 放宽，配合高赔 |
| 跨市场惩罚 | 0.85 | 同场相关性 |
| 同联赛惩罚 | 0.90 | 同联赛相关性 |
| 跨联赛奖励 | 1.10 | 降低相关性 |

### 4.4 仓位控制（核心风控）

| 赔率区间 | 单注占比 | Kelly折扣 | 最大日投注 |
|----------|----------|-----------|-----------|
| 2.0-5.0 | 0.3% | 0.20 | 3注 |
| 5.0-10.0 | 0.2% | 0.15 | 2注 |
| 10.0-20.0 | 0.1% | 0.12 | 1注 |
| > 20.0 | 0.05% | 0.10 | 1注 |

**核心逻辑**：赔率越高 → 仓位越低 → 单笔风险可控 → 长期正期望值。

### 4.5 串关轮次制回测

```
按日期分组 → 同日比赛 → 单场预测 → 串关筛选 → 组合生成 → 结算
     ↓
  下一日比赛 → ...
```

## 八、关键原则

1. **串关独立筛选**：不复用单场阈值，有自己的宽松筛选逻辑
2. **真实赔率优先**：所有回测使用Pinnacle/B365/Max/Avg真实赛前赔率
3. **联赛特化**：每个联赛独立参数，避免"一刀切"
4. **动态适应**：参数每赛季重新校准，适应市场变化
5. **风控优先**：串关仓位比单场更保守，赔率越高仓位越低
6. **赔率不设上限**：允许高赔存在，靠仓位控制风险

## 九、Phase 0-4 回测框架重构方案（待执行）

### 目标
让回测框架使用与生产环境相同的58因子引擎，确保回测结果直接反映生产表现。

### 当前问题
```
生产环境: 58因子 → logit → sigmoid → 泊松融合 → 价值计算
回测环境: Elo+DC → 直接输出概率 → 简单投票
```

回测用的是简化模型（EloDCModel、PoissonModel、FormModel、MarketModel），
没有使用生产环境的因子引擎和概率融合流程。

### 重构方案

#### 步骤1：创建回测适配器
```python
class BacktestAdapter:
    """将CSV比赛数据转换为 MatchContext，供生产引擎使用"""
    def __init__(self, league_id: str):
        self.pipeline = GameFlowPipeline(league_id)
        self.elo = EloColdStart()
    
    def predict(self, match: RealOddsMatch) -> PipelineResult:
        context = self._build_match_context(match)
        return self.pipeline.run_stages_1_5(context)
    
    def update(self, match: RealOddsMatch):
        self.elo._process_single_match(...)
```

#### 步骤2：统一数据流
```
CSV → RealOddsMatch → BacktestAdapter → MatchContext → GameFlowPipeline → PipelineResult
```

#### 步骤3：保留简化模型作为基准
简化模型（EloDCModel等）保留为"基准模型"，与生产引擎对比。

#### 步骤4：验证
- 跑同一个数据集，对比简化模型 vs 生产引擎的结果
- 确认生产引擎的回测结果合理

### 预期效果
- 回测结果 = 生产环境表现
- 因子优化可以直接在回测中验证
- 消除"回测好、实盘差"的风险
